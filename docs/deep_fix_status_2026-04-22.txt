    "Recall-first review UI + retrieval-mode telemetry"
----------------------------------------------------------------

Goal: Unblock the thesis's primary claim by making Phase 1 a real
recall event — not a recognition / re-read event. Log enough signal
that the retention regression can restrict to free-recall rows.

Schema delta on review_log:
  reveal_used BOOLEAN NOT NULL DEFAULT FALSE
  recall_mode VARCHAR(12) NULL    (free_recall | cloze | assisted)
Backfill: legacy rows marked reveal_used=TRUE, recall_mode='assisted' so
they don't silently pollute a free-recall cohort.

Backend:
  POST /api/review/submit — SubmitReviewRequest accepts reveal_used and
  recall_mode (validated against the enum; unknown → 422). Both persist
  to review_log on the same write as the grade.
  research/export includes both fields.

Frontend — ReviewSession.jsx Phase 1 layout:
  [card_type badge if action]
  study_question
  ConfidenceRating
  recall textarea
  [Reveal passage] button (only for manual/quiz/chat)
  Submit

Reveal state machine (revealState):
  hidden   (initial)   → reveal_used=false, recall_mode='free_recall'
  cloze    (1st click) → reveal_used=true,  recall_mode='cloze'
  assisted (sub-opt)   → reveal_used=true,  recall_mode='assisted'
Resets on every handleNext.

Action cards (explain / simplify / terms / summarise) NEVER get a
reveal button in Phase 1 — source surfaces only in Phase 2.

Phase 2: appended a cooler-accented .rv-source--graded panel below the
expected answer so action cards still get their text check.

Removed: ReviewModeToggle + reviewMode state + .rv-mode-* CSS.

Dev-only instrumentation:
  main.jsx exposes window.__APP_STORE__ when import.meta.env.DEV so

4. PENDING — STEP 4  (P1 leverage; P1 for Tier 1B learning graph)
   "Rhetorical-facet + topic ontology (SCIM D2 + D6)"
================================================================

Severity: P1. Not thesis-blocking, but without it the index remains
noisy and the eventual learning graph has no principled grouping.

Scope — three layers:

  (a) Layer 1 — Rhetorical Facet (SCIM D2 — peer-reviewed, do NOT invent)
      qa_pairs already has rhetorical_facet + facet_confidence (step 1).
      New work:
        - Haiku classifier at card creation. Prompt returns
          {label, confidence} over
          {objective | novelty | method | result | background | uncategorized}.
        - D6 GUARD: if confidence < 0.7 set label = 'uncategorized'.
          Better to show no badge than a wrong one.
        - Color palette (from Scim paper):
          Objective green / Novelty orange / Method blue / Result red /
          Background grey.
        - Render the facet badge in the index AND in ReviewSession Phase 1.
          The Phase 1 badge slot already exists (.rv-question-badges) — just
          append.

  (b) Layer 2 — Per-PDF Topic Ontology (5–8 items)
      New column:  pdf_documents.ontology_json JSONB NULL
      New endpoint: POST /api/pdfs/{id}/ontology
        Haiku generates 5–8 topic labels ONCE per PDF from
        abstract + TOC + first N chunks. Rate-limit 1/24h/PDF.
      Per-QA tagging becomes classification INTO this ontology
        (multi-label, ≤ 3 tags + optional "other").
      Ontology is regenerable; old classifications invalidated when it changes.

  (c) Layer 3 — Cluster Tag (derived, no schema churn on highlights)
      New column:  highlight_entries.cluster_tag VARCHAR(200) NULL
      Strategy at write time:
        If deep_section_path[0].title is non-junk per regex
          /^(arXiv|T\d|Fig|\d+\s)/i → use it.
        Else nearest ontology topic by embedding.
      Result: By-Section view "Other Passages" bucket dissolves into
      meaningful topic clusters.

FSRS due-ordering tweak (SCIM D3 "long middle"):
  When multiple cards are due, order by (due_at, section_diversity) —
  small penalty if previous review was from the same section.
  ~15-line change in review_service.py.

Backfill endpoint:
  POST /api/pdfs/{id}/recompute-facets-and-concepts
  Regenerates ontology + reclassifies all existing QAs. Run once per PDF.
  Also replaces any study_question that equals a _FALLBACK_STUDY_QUESTION
  template with a Haiku-specific rewrite.

Done-criteria (from plan §Test Plan):
  - Concepts tab label shows "Facets N · Topics M" not "Concepts34".
  - Every card has rhetorical_facet set (possibly 'uncategorized').
  - cluster_tag populated for every entry.
  - "Other Passages" bucket empty in By-Section view.

Budget estimate: 1–2 days.
